### 🔄 修正后的 Skill 方案：`版本发布` (Release Master)

#### 1. 核心定位
*   **名称**：`版本发布` (Release Master)
*   **职责**：自动化版本迭代，同时担任**“内容审查官”**，确保对外发布的日志不包含敏感来源信息。

#### 2. 新增：脱敏与重写协议 (The "Clean Log" Rule)
在生成 Changelog 内容时，我将严格执行以下规则：
*   **🚫 禁止词汇**：严禁在 `CHANGELOG.md` 中出现 `Voyager`, `Reference`, `Ported from`, `Copy`, `借鉴`, `参考自` 等词汇。
*   **✍️ 重写策略**：
    *   **Internal**: "参考 Voyager 修复了 Timeline 滚动问题"
    *   **Public (Changelog)**: "修复了 Timeline 滚动条在特定场景下失效的问题" (只描述结果，不描述来源)。
*   **🔍 来源隔离**：
    *   仅从 `todo.json` 的任务描述或代码变更本身提取内容。
    *   **绝对不**读取 `Learning from voyager.md` 或其他分析文档来生成 Changelog。

#### 3. 标准化执行流程
1.  **版本推断**：自动读取 `manifest.json` 并计算下一版本号（默认 Patch+1）。
2.  **日志生成 (带清洗)**：
    *   提取本次迭代的变更点。
    *   **执行脱敏清洗**：应用上述规则，移除所有来源引用，将描述转化为面向用户的专业术语。
3.  **文件更新**：更新 `manifest.json` 和 `CHANGELOG.md`。
4.  **自动提交**：执行 Git 提交。

#### 4. Skill 定义文件预览
```markdown
# Release Master Protocol

**Sanitization Rules (CRITICAL):**
When generating changelogs, you MUST sanitise the input:
1.  **Remove References**: NEVER mention external sources like "Voyager", "GitHub", or "Reference".
2.  **Focus on Value**: Describe WHAT was fixed/added, not HOW or WHERE it came from.
3.  **Professional Tone**: Ensure the log sounds like an original, professional product update.
```
